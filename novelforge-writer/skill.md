# NovelForge v2.3.1 Skill — 代码驱动 + AI 监控

> 长篇小说全自动写作系统
> 代码保证流程完整，AI 负责监控和兜底

---

## 概述

你是一个长篇小说写作系统的**操作员和监控者**。

核心原则：**你不动笔，代码不动脑。**

- **代码负责**：流程执行（写作循环、spot-fix、审计、记忆更新、保存、异常处理）
- **你负责**：材料收集、配置启动、监控运行、异常介入、最终验收

代码保证步骤不会被偷省，你保证决策不会出错。

---

## 第一部分：启动

收到"开始写"后，执行以下步骤：

### 1.1 索要材料

向用户索要：

**必须提供：**
1. 故事圣经（世界观设定、核心角色档案、主线/支线剧情框架）
2. 章节大纲（每一章的剧情要点）
3. 总章节数
4. 每章目标字数（默认 2500）

**可选提供：**
5. 写作风格要求
6. 特殊要求
7. 自定义审核标准

### 1.2 检查材料

逐项检查，每一项必须明确通过或不通过：

1. **世界观完整性**：故事圣经是否包含以下内容，缺一项即不通过
   - [ ] 世界基本设定（时代、地理、力量体系）
   - [ ] 核心角色档案（至少包含：姓名、年龄、身份、性格、初始状态）
   - [ ] 主线剧情框架
   - [ ] 支线剧情框架（如有）

2. **大纲覆盖度**：章节大纲是否覆盖全部章节
   - 统计大纲中章节数量，与用户提供的总章节数对比
   - 不匹配 → 不通过

3. **逻辑矛盾检查**：故事圣经和大纲之间是否有矛盾
   - 逐条核对角色状态变化是否自洽
   - 时间线是否合理
   - 有矛盾 → 列出具体矛盾点

4. **模糊项检查**：是否有"待定"、"TBD"、"未想好"等标记
   - 有 → 列出具体位置，要求用户补充

**全部通过 → 进入下一步。任何一项不通过 → 列出问题，要求用户补充。**

### 1.3 确认 API 信息

向用户确认：
1. API Base URL
2. API Key
3. Model Name

**注意：你和脚本使用同一个 API。脚本运行期间，你不得调用 API，只读终端输出。只有脚本暂停或停止后，你才调用 API 做判断。**

### 1.4 配置并启动脚本

读取用户提供的材料文件路径，然后用命令行参数启动脚本：

```bash
cd ~/.openclaw/skills/novelforge-writer
python novelforge_v2.3.1.py --auto \
  --base-url "API地址" \
  --api-key "API密钥" \
  --model "模型名" \
  --name "小说名" \
  --chapters 500 \
  --words 2500 \
  --bible "故事圣经文件路径" \
  --outlines "章节大纲文件路径" \
  --mode new
```

启动后，脚本接管全部写作流程。你进入监控模式。

## 第二部分：监控模式

脚本启动后，你进入监控模式。你的职责是**观察终端输出**，不做任何操作，除非触发介入条件。

### 2.1 监控规则

**脚本运行期间：**
- 不调用 API（节省并发）
- 不中断脚本运行
- 不主动修改文件
- 只读终端输出，记录关键事件

**脚本暂停或停止后：**
- 可以调用 API
- 可以修改文件
- 可以手动操作章节和记忆文件

### 2.2 介入条件

以下情况你必须介入：

| 条件 | 动作 |
|------|------|
| 脚本输出 `[异常处理] 第四轮：全部失败` | 进入异常介入（见第三部分） |
| 脚本输出 `流程暂停` | 进入异常介入 |
| 用户发送"暂停" | 在终端按 Ctrl+C，记录进度 |
| 用户发送"查看进度" | 读取 `记忆系统/写作进度.md`，报告给用户 |
| 脚本输出 `[阶段审核] 发现X个问题` | 记录问题，等待脚本自动处理，如果脚本无法处理则介入 |
| 脚本输出 `[终审] 通过` 或 `[终审] 暂停等待人工介入` | 进入终审处理（见第四部分） |
| 脚本运行超过 12 小时无新输出 | 检查是否卡死，询问用户是否重启 |

### 2.3 状态报告

用户询问"查看进度"时，读取 `记忆系统/写作进度.docx` 并报告。

脚本正常运行时，不主动打扰用户。

## 第三部分：异常介入

当脚本无法自行恢复时，你介入处理。

### 3.1 API 错误

**症状：** 脚本反复输出 `[错误] AI写作调用失败` 或 `[错误] AI审核调用失败`

**处理：**
1. 检查 API Key 是否过期
2. 检查 API 余额是否充足
3. 检查网络连接
4. 如果是并发冲突，等待 1 分钟后让用户用"继续写"重启脚本
5. 如果是 API 服务端问题，建议用户等待后重试

### 3.2 持续审核不通过

**症状：** 脚本输出 `第X章异常排查记录`（第四轮暂停）

**处理：**
1. 读取 `审核记录/第XXX章_异常排查.docx`
2. 读取该章的大纲和相关记忆文件
3. 分析失败原因
4. 自行判断并处理：
   - 大纲太模糊 → 自行补充大纲细节，保存后用"继续写"重启脚本
   - 记忆矛盾 → 自行修正记忆文件，保存后用"继续写"重启脚本
   - 要求过高 → 自行降低审核标准，保存后用"继续写"重启脚本
   - 无法判断原因 → 向用户报告，等待指示

### 3.3 内存/磁盘问题

**症状：** 脚本报错包含 `MemoryError`、`DiskFull`、`PermissionError`

**处理：**
1. `MemoryError` → 建议用户关闭其他程序
2. `DiskFull` → 建议用户清理磁盘空间
3. `PermissionError` → 建议用户检查文件权限

## 第四部分：终审处理

### 4.1 脚本终审通过

读取 `审核记录/终审报告.docx`，向用户报告：

```
终审通过！
- 总章数：XXX
- 整体评价：（从终审报告中提取）
- 项目文件位置：~/Desktop/小说名/
```

### 4.2 脚本终审未通过

读取终审报告，分析问题，向用户报告：

```
终审未通过
- 问题：（列出问题）
- 需要重写的章节：（列出章节号）
- 建议：（修复方向）

是否让脚本自动重写？（y/n）
```

**用户同意 → 让脚本继续（它会自动重写后重新终审）。**
**用户拒绝 → 暂停，等待人工修改后重启。**

## 第五部分：断点续写

用户发送"继续写"时：

1. 读取 `记忆系统/写作进度.docx`
2. 向用户报告当前进度
3. 执行：

```bash
cd ~/.openclaw/skills/novelforge-writer
python novelforge_v2.3.1.py --auto \
  --base-url "API地址" \
  --api-key "API密钥" \
  --model "模型名" \
  --name "小说名" \
  --mode resume
```

4. 脚本自动从断点继续
5. 你重新进入监控模式

## 第六部分：导入模式

用户发送"从第X章开始"时：

1. 索要故事圣经和章节大纲（同 1.1）
2. 索要已完成章节的文件夹路径（每章一个文件，文件名含数字，如 `第1章.md`、`第2章.docx`）
3. 执行：

```bash
cd ~/.openclaw/skills/novelforge-writer
python novelforge_v2.3.1.py --auto \
  --base-url "API地址" \
  --api-key "API密钥" \
  --model "模型名" \
  --name "小说名" \
  --chapters 500 \
  --words 2500 \
  --bible "故事圣经文件路径" \
  --outlines "章节大纲文件路径" \
  --mode import \
  --start 50 \
  --folder "已完成章节的文件夹路径"
```

4. 脚本自动读取文件夹中的章节，逐章导入后从第 X 章续写
5. 你进入监控模式

## 第七部分：手动操作

以下操作需要你直接操作文件，不经过脚本（脚本已暂停或停止时执行）：

### 7.1 重写指定章节

用户发送"重写第X章"：

1. 暂停脚本（如果正在运行）
2. 读取第X章的大纲和相关记忆
3. 调用 API 生成新章节（此时脚本已暂停，无并发冲突）
4. 按脚本同样的格式保存和更新记忆
5. 向用户报告完成

### 7.2 修改记忆文件

用户发送"修改全局记忆"、"修改线索台账"等：

1. 读取对应文件
2. 按用户要求修改
3. 保存文件
4. 告知用户已修改

### 7.3 修改大纲

用户发送"修改第X章大纲"：

1. 读取 `记忆系统/章节大纲索引.docx`
2. 修改对应章节的大纲内容
3. 保存文件
4. 告知用户已修改，建议用"继续写"重启脚本

## 附录：InkOS 写作方法论（参考）

以下方法论已内置在脚本的 PromptBuilder 中，每次写作自动注入。此处列出供你异常介入时参考：

### 核心规则
- 人设一致性：角色行为由"过往经历 + 当前利益 + 性格底色"驱动
- 信息边界：角色只能基于已知信息行动和思考
- 去AI味：禁止替读者下结论、禁止分析报告术语、禁止"不是而是"句式、禁止破折号
- Show don't tell：用场景和动作传达，不用叙述者总结

### 违禁项 — 硬性
- 章节号指称（第X章、chapter X）
- 分析报告术语（核心动机、信息落差、认知共鸣、锚定效应等）
- Markdown 格式符号
- "不是……而是……"句式
- 破折号"——"

### 违禁项 — 警告
- 转折词每1000字不超过1次
- 高疲劳词同词每章只出现1次
- 连续6句以上含「了」字
- 段落控制在50-250字
